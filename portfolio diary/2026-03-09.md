# 2026-03-09

## What changed
- Implemented auth callback hardening to prevent token exchange race conditions that were causing `POST /oauth2/token` requests to hit the app domain and return `404`.
- Added explicit Cognito config guardrails and client-side auth diagnostics (safe metadata only, no secrets/tokens).
- Fixed Chime video cleanup to unbind by tile ID (not by HTML video element), which addresses Safari warnings about unknown tile IDs.
- Added frontend room/media diagnostics for socket and Chime lifecycle events (`room_joined`, `video_ready`, `game_started`, tile update/remove, media stop/fail).
- Added temporary room debug status line in the game header to show `connected players`, `game active`, `my color`, and `my turn` during two-device repro.
- Added backend observability logs around room join and Chime setup: room status transitions, meeting create start/success/failure, attendee creation, and `VIDEO_READY` broadcast.
- Added backend game start transition log for correlation with first-move gating tests.

## How it was done
- Updated frontend files:
  - `app/frontend/src/context/AuthContext.jsx`
  - `app/frontend/src/pages/AuthCallbackPage.jsx`
  - `app/frontend/src/pages/RoomPage.jsx`
- Updated backend file:
  - `app/backend/src/websocket/handler.js`
- Validation commands:
  - `npm --prefix app/frontend run test`
  - `npm --prefix app/backend run test`
  - `npm --prefix app/frontend run build`

## Why these decisions were made
- The highest-impact issue was auth instability during callback. Stabilizing callback config resolution first reduces downstream websocket churn and room state instability.
- Chime unbind warnings in Safari directly matched incorrect API usage (`unbindVideoElement` called with element/null). Fixing this removes noisy warnings and avoids cleanup inconsistencies.
- Added targeted, structured diagnostics in both frontend and backend to make cross-device repro actionable in CloudWatch and browser console without exposing sensitive data.
- Temporary game-state debug labels accelerate validation of move gating (`game_started` must be received before first legal move expectations).

## Risks and tradeoffs
- Added debug logging increases console/log volume temporarily; this is intentional for active incident-style troubleshooting.
- Temporary room debug status copy is not final UX polish and should be removed or hidden behind a debug flag after verification is complete.

## Additional Session Update (Canonical Host Redirect to App Subdomain)

### What changed
- Implemented Terraform canonical-host strategy to make `https://app.chess-chat.com` the single runtime/auth origin.
- Added ALB HTTPS host-routing rules:
  - `Host: chess-chat.com` => `301` redirect to `https://app.chess-chat.com#{path}?#{query}`
  - `Host: app.chess-chat.com` => forward to app target group
  - Default HTTPS action => fixed `404` response for unknown hosts
- Updated Terraform domain derivation:
  - canonical host is now computed as `app.<root_domain>` when present
  - Cognito derived callback/logout URLs now use canonical host only
- Updated runtime env desired state:
  - `APP_DOMAIN` from `https://chess-chat.com` to `https://app.chess-chat.com`
- Updated landing copy to reduce auth UX confusion:
  - Added note that username is chosen after sign-in in Lobby (not in Cognito Hosted UI).

### How it was done
- Terraform files:
  - `terraform/main.tf`
  - `terraform/modules/alb/main.tf`
  - `terraform/modules/alb/variables.tf`
  - `terraform/environments/dev/terraform.tfvars`
- Frontend file:
  - `app/frontend/src/pages/LandingPage.jsx`
- Documentation updates:
  - `DOCS/PORTFOLIO_BUILD_PLAYBOOK.md`
  - `DOCS/UI_DESIGN_GUIDE.md`
  - `infra/RESOURCE_REGISTRY.md`

### Why decisions were made
- OAuth `state` relies on origin-scoped browser storage; mixed apex/subdomain auth paths caused `OAuth state mismatch`.
- A single canonical host removes cross-origin callback ambiguity and makes login deterministic.
- Keeping apex DNS alias but redirecting at ALB preserves user-friendly entrypoint while enforcing one auth path.
