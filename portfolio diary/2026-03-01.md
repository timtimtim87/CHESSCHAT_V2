## Problems Faced and How We Solved Them
- Problem: Terraform plan failed when Redis SG rules used `for_each = toset(var.allowed_security_group_ids)` and the list included ECS SG IDs only known after apply.
  - Why it happened: `for_each` keys must be known at plan time; unknown resource-derived values cannot define dynamic instance keys.
  - Fix: Split ElastiCache ingress handling into:
    - static list-based rules (`allowed_security_group_ids`)
    - dedicated ECS SG rule with `count` gated by a known boolean (`enable_ecs_service_ingress`) and value passed via `ecs_service_security_group_id`.
  - Result: Plan became deterministic and apply-safe.

- Problem: Targeted apply for only ECR failed because Terraform had module address moves (`module.ecs` -> `module.ecs_identity`) that were not included in target scope.
  - Why it happened: Terraform requires moved-address reconciliation in the same plan graph; narrow `-target` excluded required moved objects.
  - Fix: Switched to full apply sequence with temporary service scale guard:
    - set `ecs_service_desired_count = 0`
    - apply full stack changes (including move reconciliation and infra creation)
    - push image
    - set desired count back to `1` and apply again.
  - Result: Migration and resource creation completed cleanly without task startup failures.

- Problem: ECS tasks failed with `CannotPullContainerError` (manifest did not match `linux/amd64`).
  - Why it happened: initial pushed image manifest did not include the architecture required by task runtime.
  - Fix: rebuilt and pushed the bootstrap image with explicit platform:
    - `docker buildx build --platform linux/amd64 --push ...`
    - forced ECS redeploy via `aws ecs update-service --force-new-deployment`.
  - Result: ECS service converged to `running=1`, `desired=1`, `pending=0`.
